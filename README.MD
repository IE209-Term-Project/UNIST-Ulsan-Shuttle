**AI-Based Demand-Responsive Shuttle Optimization for UNIST–Ulsan Station Mobility** (**UNIST–울산역 수요반응형 보조 셔틀 최적화 및 개인화 이동 에이전트 설계)**

: 고정 셔틀 시간표를 먼저 최적화하고, 그 위에 예약 수요와 실시간 교통 상황을 반영해 운행 여부·출발 알림·대체 경로를 조정하는 에이전트 시스템

- 문제 정의

현재 UNIST 학생들은 울산역 KTX 이용 시 기존 버스 시간과 열차 시간의 불일치로 인해 긴 대기시간, 과도한 조기 출발, 택시 비용 지출 등의 이동에서의 불편함을 경험한다. 본 프로젝트는 기존 버스 노선을 변경하지 않고, KTX 수요가 집중되는 시간대에만 예약 기반 보조 셔틀을 투입하여 최소 운영 비용으로 학생 대기시간과 이동 불확실성을 줄이는 AI 기반 수요반응형 운영 시스템을 설계한다.

- 실제 해결 방안

완전 실시간으로 아무 때나 부르는 게 아니라, KTX 도착·출발 시간 기준으로 후보 셔틀 시간대 생성

1. **먼저 고정 후보 시간대를 만든다**

예를 들면:

| **상황** | **후보 셔틀 시간** |
| --- | --- |
| UNIST → 울산역 | KTX 출발 50~60분 전 |
| 울산역 → UNIST | KTX 도착 후 10~20분 이내 |
| 금요일 저녁 | 16:00~21:00 집중 |
| 일요일 저녁 | 18:00~22:00 집중 |
| 공휴일 전날 | 추가 후보 시간 생성 |

이 후보 시간대는 KTX 시간표, 513번 버스 시간표, 학생 설문 데이터를 기반으로 만들면 돼.

1. **그다음 최적화 모델로 “운행할 시간”을 고른다 → 최소화? 필요한 것만 고르기**

먼저 KTX 시간표와 학생 이동 패턴을 기준으로 ‘운행 후보 시간대’를 만들고, 그중에서 고정 셔틀이 아닌 수요반응형 후보 슬롯에 대해 최적화 모델이 운행 여부를 결정한다.

- 고정 운영 셔틀 ->
- 수요반응형 후보 셔틀 -> 결정 변수로 운행 결정

KTX 시간표 + 기존 버스 시간표 분석

↓

운행 후보 시간대 생성

↓

후보 시간대를 두 그룹으로 분류

↓

1) 수요가 확실한 시간대 → 고정 셔틀

2) 수요가 불확실한 시간대 → 최적화 모델로 운행 여부 결정

↓

Agent가 학생에게 셔틀 확정 / 미운행 / 대체 경로 알림

1. **에이전트는 여기서 강하게 활용한다**

에이전트는 단순히 “알림 보내기”만 하면 약해. 다음처럼 역할을 나누면 좋아.

| **Agent** | **역할** |
| 실시간 Data Agent | API Key 를 활용한 실시간 정보 제공|
| Demand Prediction Agent | 금요일 저녁, 일요일 밤, 시험기간, 공휴일 전후 수요 예측 |
| Notification 및 Recommnedation Agent | 학생 개인 일정/KTX 시간에 맞춰 출발 알림, 대체 경로, 지연 경고 제공 |
| Report Agent | 관리자용 운영 리포트 생성: 몇 회 운행, 몇 명 수송, 대기시간 얼마나 감소했는지 |

여기서 핵심은 **LLM이 직접 최적화 계산을 하는 게 아니라, Python/OR-Tools가 계산한 결과를 에이전트가 해석하고 의사결정 메시지로 바꾸는 구조**야.

Q1. 셔틀 수요가 없어서 셔틀이 운영되지 않으면 손해가 크지 않아?

아무 때나 부르는 셔틀이 아님! KTX 시간표와 기존 버스 시간표를 기준으로 셔틀 후보 시간대를 만들고, 예약 수요나 예측 수요가 일정 기준을 넘을 때만 운행을 확정한다.

| **유형** | **설명** |
| --- | --- |
| Fixed Peak Shuttle | 금요일 저녁, 일요일 밤처럼 수요가 확실한 시간대는 고정 운행 |
| Conditional Shuttle | 수요가 애매한 시간대는 예약자가 기준 이상일 때만 운행 |
| No Shuttle | 기준 미달이면 기존 버스, 택시 쉐어, 대체 경로 안내 |

셔틀 운행 편익 ≥ 셔틀 1회 운행 비용 일 때만 운영

결국에, Fixed + Conditional Hybrid Model

실제 사례) 인천 I-MOD, 부산 타바라

- UNIST에서는?
1. **Fixed Peak Block**

수요가 확실한 시간대에는 셔틀을 고정 배치한다.

예시:

- 금요일 오후: UNIST → 울산역
- 일요일 저녁: 울산역 → UNIST
- 공휴일 전날/마지막 날
- 개강종강 기간

이때 기사님은 해당 시간 블록에 근무하고, 셔틀은 정해진 후보 시간에 맞춰 반복 운행한다.

1. **Reservation-Based Conditional Dispatch**

수요가 불확실한 시간대는 예약 기준을 넘을 때만 운행한다.

예시:

- 평일 낮
- 토요일 오전
- 시험기간 중 특수 시간대
- KTX 도착이 몰리는 특정 시간

이때는 기사님이 계속 대기하는 게 아니라, **예약이 기준 이상일 때만 운행 계약 또는 배차가 발생**하는 구조로 설정한다.

1. **Alternative Recommendation**

수요가 기준 미달이면 에이전트가 대체안을 제공한다.

예시:

- “513번 이용 시 예상 도착 시간”
- “5005번 환승 경로”
- “같은 KTX를 타는 학생 3명과 택시 쉐어 가능”
- “교통 정체로 인해 15분 조기 출발 필요”
